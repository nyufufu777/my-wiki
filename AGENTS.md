# my-wiki の運用ルール

## GitHub

- `main` へ直接 push しない。変更は `agent/` 接頭辞のブランチで行い、必ず PR を作成して、ユーザーにマージを依頼する。
- 複数記事にまたがる追加、情報設計の変更、UI の変更など大きめの作業は、実装前に GitHub Issue を作る。PR 本文に `Closes #<Issue番号>` を書く。
- 作業ツリーに関係ない変更があるときは、それを stage しない。

## 記事

- 学習記事は `docs/` に置き、左ナビは「ジャンル → 記事」の階層で整理する。
- トップページは記事一覧への入口とし、記事本文を重複して置かない。
- 初心者が具体像をつかめるよう、概念だけで終わらせず、必要に応じてコード・コマンド・通信例を添える。
- 製品固有の説明は一般概念の記事と分ける。

## 図

- 依存関係、通信手順、状態遷移、分岐が文章だけでは追いにくいときに Mermaid を使う。
- 装飾目的の図、本文をそのまま言い換えるだけの図は追加しない。
- 図の直後には、図から読むべき点を短く文章で補足する。

## 検証

- 設定や記事を変更したら、可能な限り `python -m mkdocs build --strict` を実行する。
- ナビゲーションやスタイルを変更したら、PC とモバイルで主要な導線を確認する。
