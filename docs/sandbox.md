# Sandbox入門：隔離・権限・調査の出発点

## sandbox は「安全な実験室」

sandbox（サンドボックス）は、プログラムを**許可された範囲だけで動かすための隔離環境**。子どもが砂場の外へ勝手に行かないようにする仕組み、くらいに考えればいい。

Codex にコマンド実行やファイル編集を頼むとき、sandbox は「どこを読めるか」「どこに書けるか」「外部ネットワークへ接続できるか」を制限する。目的は、作業対象外のファイル、秘密情報、OS 全体、外部サービスを意図せず触らないようにすること。

```text
通常のPC環境
  プログラム → ディスク全体・ネットワークへ到達できる

sandbox環境
  プログラム → 許可された作業フォルダ・許可された通信先だけへ到達できる
```

大事なのは、sandbox は「壊れない魔法の箱」ではないこと。許可範囲を広げればできることも増えるけれど、その分だけ影響範囲も増える。だから、必要最小限にするのが基本よ。

## 何が隔離されるのか

sandbox で特に見るのは、この3つ。

| 対象 | 確認すること | 例 |
| --- | --- | --- |
| ファイルシステム | どこを読める・書けるか | プロジェクト内は編集可、ホーム直下は不可 |
| ネットワーク | 外部へ接続できるか | パッケージのダウンロード、GitHub API、社内 API |
| 実行権限 | どのコマンドがそのまま動くか | テストは可、削除や Git 設定変更は承認が必要 |

たとえば、次の依頼は同じ「コマンドを実行する」でも性質が違う。

```text
npm test
```

これは通常、プロジェクト内のテストを走らせるだけ。書き込みがなければ影響範囲は小さい。

```text
pip install -r requirements.txt
```

これは外部のパッケージ配布元へ接続し、依存関係を取得する。ネットワーク制限に引っかかるかもしれない。

```text
git push origin main
```

これは外部リポジトリの状態を変える。ファイル操作だけでなく、外部サービスへの書き込みでもある。だから確認や承認が必要になりやすい。

## 「拒否された」は失敗とは限らない

sandbox で操作が拒否されても、コマンドやコードが間違っているとは限らない。単に「その操作の種類には、まだ許可がない」という意味の場合がある。

例として、プロジェクト内のファイルを編集するつもりで次を実行したとする。

```powershell
Set-Content .\notes.txt "hello"
```

作業フォルダが書き込み可能なら成功する。一方、OS の設定ファイルを変えようとする操作は、たとえ文法が正しくても拒否されうる。

```powershell
Set-Content C:\Windows\System32\example.txt "hello"
```

重要なのは、拒否を回避するために対象を曖昧に広げないこと。「必要なファイルは何か」「外部通信は本当に必要か」を切り分けて、狭い許可を要求する。

## Codex では sandbox と承認は別のもの

似ているけれど、次の2つは役割が違う。

| 仕組み | 答える問い | 例 |
| --- | --- | --- |
| sandbox | その操作は技術的にどこまで可能か | 作業フォルダ以外への書き込みを遮断する |
| 承認ポリシー | 実行前に人や自動レビューの確認が必要か | `git push` の前に確認を求める |

つまり「sandbox 内で実行できる」ことと「確認なしに実行してよい」ことは同じじゃない。読み取り専用の操作でも、環境によっては承認対象になることがあるし、承認された操作を sandbox の外で限定的に実行することもある。

Codex の設定では、承認の厳しさを `approval_policy`、sandbox の範囲を `sandbox_mode` として別々に扱う。代表的な考え方は次の通り。

```toml
approval_policy = "on-request"
sandbox_mode = "workspace-write"

[sandbox_workspace_write]
writable_roots = ["/path/to/extra-writable-directory"]
network_access = false
```

この例なら、基本は作業領域に書き込めるが、追加で許可したディレクトリ以外は対象外。外部ネットワークも既定では使わない、という意図になる。実際に有効な設定は、アプリ・CLI・組織の管理ポリシーによって変わる。

## `workspace-write` を具体的に考える

`workspace-write` は「何でも書ける」ではなく、**指定されたワークスペースと writable roots に限定して書ける**という意味。

```text
workspace: /work/my-app
writable roots: /work/my-app, /tmp/task-output

編集できる例
  /work/my-app/src/app.ts
  /tmp/task-output/report.txt

編集できない例
  /etc/hosts
  /home/user/.ssh/config
```

しかも、ワークスペース内でも `.git/` や `.codex/` のような管理用ディレクトリは読み取り専用にされることがある。だから、ソースコードの編集は成功するのに、Git の設定変更やコミットが承認を求める、という状況はあり得る。

## ネットワーク制限があると何が起きるか

ネットワークを閉じると、ローカルの調査・編集・テストはできても、次のような操作は止まる。

```text
外部パッケージを取得する      → npm install / pip install が失敗する
リモートリポジトリを読む      → git fetch が失敗する
外部 API を呼ぶ               → curl や SDK の通信が失敗する
```

これは不具合ではなく、意図された境界。必要なら、操作ごとにネットワーク利用の承認を求める。依存関係を追加するときは、パッケージ名・取得先・なぜ必要かを先に確認するのが安全ね。

## 依頼を安全に分ける例

「ライブラリを入れて、テストして、GitHub に公開して」は、一つの作業に見えても性質が3種類ある。

```text
1. npm install             → 外部ネットワーク + ローカルへの書き込み
2. npm test                → ローカル実行（テスト出力やキャッシュの書き込みの可能性）
3. git push origin branch  → 外部サービスの状態変更
```

このように分けて考えると、どこで許可が要るのか、失敗したときに何を確認すべきかがはっきりする。特に `git push` は、sandbox の話だけでなく「誰が公開を承認するか」という運用の話でもある。

## 後で Codex 環境を調査するときの観点

次の調査では、実際の環境設定をこの順で確認するといい。

1. 現在の `sandbox_mode` は何か
2. 読み取り・書き込みが許可されたパスはどこか
3. `.git`、`.codex`、一時ディレクトリはどう扱われるか
4. 外部ネットワークは既定で使えるか。承認後に使えるか
5. `approval_policy` と自動レビューは何か
6. 管理ポリシーがローカル設定より優先していないか

確認用のコマンドも、いきなり書き換えるものを使わない。まずは現在地と権限を読むだけの操作から始める。

```powershell
Get-Location
Get-ChildItem -Force
git status --short
git remote -v
```

その後に、作業フォルダ内へ小さな一時ファイルを作る、外部へ到達できるかを確認する、と段階を踏む。対象を明示した小さな検証なら、失敗しても影響は限定される。

## ここまでの要点

- sandbox はファイル・ネットワーク・実行範囲を狭める安全境界
- 拒否は「コマンドが間違い」ではなく「許可範囲外」の場合がある
- sandbox と承認ポリシーは別物
- `workspace-write` は無制限の書き込み権限ではない
- 実環境の設定は変わり得るので、将来は読み取り中心の調査で確かめる
