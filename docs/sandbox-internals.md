# Sandboxの内部：OSが境界を作る仕組み

sandbox は一つの機能名ではない。OS が持つ複数の仕組みを組み合わせて、「このプロセスにはここまで」という境界を作る。

```text
process
  ├─ どのユーザーIDで動くか
  ├─ どのファイルパスを見られるか
  ├─ どのシステムコールを許すか
  ├─ CPU・メモリをどこまで使えるか
  └─ どのネットワークへ出られるか
```

## プロセスはOSから見た「実行中のプログラム」

プログラムを起動すると、OS はプロセスを作る。プロセスごとに仮想メモリ、開いているファイル、環境変数、ユーザーID、プロセスIDがある。sandbox はまず、この単位を隔離する。

同じマシンで動く二つのプロセスでも、通常は互いのメモリを直接読めない。OS の仮想メモリとページ保護が境界になっているから。sandbox はこの既存の境界へ、追加の制限を重ねる。

## UID/GIDとファイルパーミッション

Unix 系 OS では、各ファイルに所有者と読み・書き・実行の権限がある。

```text
-rw-------  user  secret.txt
```

この例は owner だけが読めて書ける。sandbox では専用ユーザーで実行し、ホストの重要ファイルをそのユーザーから読めなくする方法が基本になる。

ただしパーミッションだけでは「見えているパス」を減らせない。読めない `/etc/shadow` があっても、ファイル名や周辺の構成は見える場合がある。そこで次のファイルシステム境界も使う。

## ファイルシステム境界：見える世界を変える

chroot、mount namespace、コンテナのレイヤーファイルシステムなどは、プロセスから見えるディレクトリツリーを狭めるために使われる。

```text
host の /                         sandbox から見える /
├─ home/                          ├─ workspace/
├─ etc/                           ├─ tmp/
├─ var/                           └─ usr/（必要最小限）
└─ workspace/
```

重要なのは「ホスト上の絶対パス」と「sandbox 内で見える絶対パス」は一致し得るが、同じ実体とは限らないこと。`/tmp` が専用の一時領域へマウントされている場合、sandbox の `/tmp/a.txt` はホストの `/tmp/a.txt` ではない。

## システムコールを絞る

アプリがファイルを開く、プロセスを作る、ネットワークへ接続する操作は、最終的に OS の system call になる。Linux では seccomp のような仕組みで、使える system call を許可リストにできる。

```text
openat()   → ファイルを開く
connect()  → ネットワーク接続
clone()    → プロセス/スレッド作成
mount()    → ファイルシステムをマウント
```

たとえば `mount()` を禁止すれば、sandbox 内のプロセスが勝手に新しいディスクを見せることを防げる。危険なのはコマンド名ではなく、最終的に OS へ何を頼むか。ここが OS レベルで考える価値よ。

## ネットワーク境界：パケットをどこへ出せるか

ネットワークもプロセスにとっては system call から始まる。`connect()` が許可されても、名前空間、ファイアウォール、プロキシ、送信先許可リストで次の境界を作れる。

```text
process → connect()
        → network namespace
        → firewall / egress policy
        → proxy（ある場合）
        → Internet
```

「ネットワークなし」は DNS だけを止める意味ではない。IP を直接指定しても出られないよう、出口そのものを閉じる必要がある。逆に許可するなら、`registry.npmjs.org` だけ、社内 API だけ、のように送信先を狭める。

## リソース制限：暴走してもホストを巻き込まない

CPU やメモリを無制限に使えると、無限ループや巨大ビルドで他の作業まで止まる。cgroups や OS の resource limit で上限を置く。

```text
CPU      : 使える時間・コア数を制限
memory   : 上限を超えたら終了または割り当て失敗
process  : 子プロセス数を制限
disk     : 一時領域や書き込み量を制限
```

`npm test` が落ちたとき、コードの失敗とは限らない。「メモリ上限で kill された」「ディスク容量が尽きた」も sandbox 環境ではあり得る。終了コード、OS ログ、使用量を分けて見る。

## sandboxは重ねるほど強くなる

一つの仕組みだけでは穴が残る。専用ユーザーだけなら見えるファイルが多い。ファイルシステムを分けてもネットワークが無制限なら外部へ送れる。だから実際は複数層を重ねる。

```text
ユーザーID
  + ファイルシステム境界
  + system call 制限
  + ネットワーク出口制御
  + CPU/メモリ上限
  = sandbox
```

## 調査のときに見る順番

1. どのユーザー・プロセスとして動いているか
2. 現在地と親ディレクトリに何が見えるか
3. 読み取り・書き込みできるパスはどこか
4. 外部 DNS、IP、HTTPS のどこで通信が止まるか
5. CPU、メモリ、ディスク、プロセス数の上限はあるか
6. sandbox 外での実行には誰の承認が必要か

この順なら、拒否された原因を「権限」「パス」「ネットワーク」「資源」のどれかへ絞りやすい。Codex における具体的な設定の話は、別ページの [Codexのsandboxと承認](codex-sandbox.md) に置く。
